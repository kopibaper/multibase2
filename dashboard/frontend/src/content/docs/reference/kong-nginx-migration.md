# 🔀 Kong → Nginx Gateway Migration

**Status:** ✅ Complete  
**Branch:** `cloud-version`

---

## Summary

Replaced all per-tenant Kong API gateways with a single shared Nginx reverse proxy container (`multibase-nginx-gateway`). This eliminates the largest RAM consumer in the classic stack.

### Result

| Metric | Before (Kong) | After (Nginx) |
|--------|--------------|---------------|
| Gateway containers | 1 per tenant | 1 total |
| RAM per gateway | ~1.4 GiB | ~50 MiB total |
| Config format | Declarative YAML | Generated `.conf` |
| Reload method | `kong reload` | `nginx -s reload` |
| DNS resolution | Built-in | `resolver 127.0.0.11` |

---

## Key Technical Decisions

1. **Deferred DNS** (`set $var; proxy_pass http://$var`) — Nginx resolves upstream at request time, not at startup. This prevents crashes when tenant containers aren't running yet.

2. **`resolver 127.0.0.11`** — Docker's embedded DNS server. Required for deferred DNS to work inside Docker networks.

3. **`map_hash_bucket_size 512`** — Increased from default 64 to handle many `map` directives across multiple tenants.

4. **Auto-generation via `setup_shared.py`** — Nginx configs are generated from Python templates on `create` and `start`, not hand-written. This ensures consistency and prevents drift.

5. **Hot reload** — `docker exec multibase-nginx-gateway nginx -s reload` is called after config changes. Zero downtime for existing connections.

## Architecture

```
Client Request
    │
    ▼
┌─────────────────────────┐
│  multibase-nginx-gateway │  (Port 8000)
│  Single shared container │
└─────────┬───────────────┘
          │  Routes by tenant prefix/domain
          ▼
    ┌─────────────┐
    │ Tenant Auth  │  /auth/v1/*
    │ Tenant REST  │  /rest/v1/*
    │ Tenant RT    │  /realtime/v1/*
    │ Tenant Store │  /storage/v1/*
    │ Tenant Edge  │  /functions/v1/*
    └─────────────┘
```

## Generated Config Structure

Each tenant gets a config file at `nginx/sites-enabled/{tenant}.conf`:

```nginx
# Auto-generated by setup_shared.py — DO NOT EDIT MANUALLY

map $request_uri $tenant_rest_upstream {
    default http://{tenant}-rest:3000;
}

map $request_uri $tenant_auth_upstream {
    default http://{tenant}-auth:9999;
}

server {
    listen 8000;
    server_name {tenant}.localhost {tenant}.example.com;

    # REST API
    location /rest/v1/ {
        set $upstream $tenant_rest_upstream;
        proxy_pass $upstream;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # Auth
    location /auth/v1/ {
        set $upstream $tenant_auth_upstream;
        proxy_pass $upstream;
        proxy_set_header Host $host;
    }

    # Realtime WebSocket
    location /realtime/v1/ {
        set $upstream http://{tenant}-realtime:4000;
        proxy_pass $upstream;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }

    # Storage
    location /storage/v1/ {
        set $upstream http://{tenant}-storage:5000;
        proxy_pass $upstream;
        client_max_body_size 50M;
    }

    # Edge Functions
    location /functions/v1/ {
        set $upstream http://{tenant}-edge-functions:9000;
        proxy_pass $upstream;
    }
}
```

## How To Add a New Tenant

```bash
python setup_shared.py create <tenant-name>
```

This automatically:
1. Creates the 5-container tenant stack
2. Generates `nginx/sites-enabled/<tenant-name>.conf`
3. Reloads the nginx gateway

No manual nginx configuration needed.

## How To Add a New Route

1. Edit the nginx config template in `setup_shared.py` (function `generate_nginx_config`)
2. Re-run `python setup_shared.py start` to regenerate all configs
3. Or manually reload: `docker exec multibase-nginx-gateway nginx -s reload`

## How To Debug Routing Issues

```bash
# Check nginx config syntax
docker exec multibase-nginx-gateway nginx -t

# View nginx error log
docker exec multibase-nginx-gateway cat /var/log/nginx/error.log

# Test upstream resolution
docker exec multibase-nginx-gateway nslookup <tenant>-rest 127.0.0.11
```

## Backward Compatibility

- **API URLs unchanged**: Same paths (`/rest/v1/`, `/auth/v1/`, etc.)
- **Same gateway port**: `:8000` (configurable via `SHARED_GATEWAY_PORT`)
- **Client SDKs**: No changes needed — Supabase JS/Python clients work as before
- **Environment variables**: `SHARED_KONG_HTTP_PORT` renamed to `SHARED_GATEWAY_PORT`
