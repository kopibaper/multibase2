# 🔀 Kong → Nginx Migration
description: Migration reference from per-tenant Kong to shared Nginx gateway

**Status:** ✅ Complete  
**Branch:** `cloud-version`

> Replaced all per-tenant Kong API gateways with a single shared Nginx reverse proxy container. This eliminates the largest RAM consumer in the classic stack.

---

## Migration Result

**Status:** ✅ Implemented

### Description

One shared Nginx container (`multibase-nginx-gateway`) now handles routing for all tenants, replacing N individual Kong containers.

### Before vs. After

- **Gateway Containers**
  - Before: 1 Kong per tenant
  - After: 1 Nginx total
- **RAM per Gateway**
  - Before: ~1.4 GiB (Kong)
  - After: ~50 MiB total (Nginx)
- **Config Format**
  - Before: Declarative YAML (`kong.yml`)
  - After: Generated `.conf` files
- **Reload Method**
  - Before: `kong reload`
  - After: `nginx -s reload`
- **DNS Resolution**
  - Before: Kong built-in
  - After: `resolver 127.0.0.11`

## 🏗️ Key Technical Decisions

**Status:** ✅ Reference

### Description

Important architectural choices made during the migration and the reasoning behind them.

### Decisions

- **Deferred DNS Resolution**
  - Uses `set $var; proxy_pass http://$var` pattern
  - Nginx resolves upstream at request time, not at startup
  - Prevents crashes when tenant containers aren't running yet
- **Docker DNS** (`resolver 127.0.0.11`)
  - Docker's embedded DNS server
  - Required for deferred DNS to work inside Docker networks
  - Set with `valid=10s` for cache refresh
- **Map Hash Bucket Size** (`map_hash_bucket_size 512`)
  - Increased from default 64
  - Required to handle many `map` directives across multiple tenants
- **Auto-Generation via `setup_shared.py`**
  - Nginx configs generated from Python templates on `create` and `start`
  - Ensures consistency and prevents config drift
  - No manual editing required
- **Hot Reload**
  - `docker exec multibase-nginx-gateway nginx -s reload`
  - Zero downtime for existing connections
  - Called automatically after config changes

## 🔌 Routing Architecture

**Status:** ✅ Implemented

### Description

The gateway routes requests to the correct tenant containers based on server name and URL path.

### Request Flow

```
Client Request
    │
    ▼
┌─────────────────────────┐
│  multibase-nginx-gateway │  (Port 8000)
│  Single shared container │
└─────────┬───────────────┘
          │  Routes by tenant prefix/domain
          ▼
    ┌─────────────┐
    │ Tenant Auth  │  /auth/v1/*
    │ Tenant REST  │  /rest/v1/*
    │ Tenant RT    │  /realtime/v1/*
    │ Tenant Store │  /storage/v1/*
    │ Tenant Edge  │  /functions/v1/*
    └─────────────┘
```

### Route Mapping

- **`/rest/v1/*`** → `{tenant}-rest:3000` (PostgREST)
- **`/auth/v1/*`** → `{tenant}-auth:9999` (GoTrue)
- **`/realtime/v1/*`** → `{tenant}-realtime:4000` (WebSocket upgrade)
- **`/storage/v1/*`** → `{tenant}-storage:5000` (max 50MB upload)
- **`/functions/v1/*`** → `{tenant}-edge-functions:9000` (Deno)

## 📝 Config Structure

**Status:** ✅ Reference

### Description

Each tenant gets an auto-generated config file at `nginx/sites-enabled/{tenant}.conf`.

### Example Config

```nginx
# Auto-generated by setup_shared.py — DO NOT EDIT MANUALLY

map $request_uri $tenant_rest_upstream {
    default http://{tenant}-rest:3000;
}

server {
    listen 8000;
    server_name {tenant}.localhost {tenant}.example.com;

    # REST API
    location /rest/v1/ {
        set $upstream $tenant_rest_upstream;
        proxy_pass $upstream;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # Auth
    location /auth/v1/ {
        set $upstream http://{tenant}-auth:9999;
        proxy_pass $upstream;
    }

    # Realtime (WebSocket)
    location /realtime/v1/ {
        set $upstream http://{tenant}-realtime:4000;
        proxy_pass $upstream;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }

    # Storage
    location /storage/v1/ {
        set $upstream http://{tenant}-storage:5000;
        proxy_pass $upstream;
        client_max_body_size 50M;
    }

    # Edge Functions
    location /functions/v1/ {
        set $upstream http://{tenant}-edge-functions:9000;
        proxy_pass $upstream;
    }
}
```

## 🛠️ Common Operations

### Add a New Tenant

```bash
python setup_shared.py create <tenant-name>
```

**What it does:**

1. ✅ Creates the 5-container tenant stack
2. ✅ Generates `nginx/sites-enabled/<tenant-name>.conf`
3. ✅ Reloads the nginx gateway

No manual nginx configuration needed.

### Add a New Route

1. Edit the template in `setup_shared.py` (function `generate_nginx_config`)
2. Re-run `python setup_shared.py start` to regenerate all configs
3. Or manually reload: `docker exec multibase-nginx-gateway nginx -s reload`

### Debug Routing Issues

```bash
# Check nginx config syntax
docker exec multibase-nginx-gateway nginx -t

# View nginx error log
docker exec multibase-nginx-gateway cat /var/log/nginx/error.log

# Test upstream resolution
docker exec multibase-nginx-gateway nslookup <tenant>-rest 127.0.0.11
```

## ✅ Backward Compatibility

### Description

The migration is fully backward-compatible. No client-side changes are required.

### What Stayed the Same

- **API URLs** — Same paths (`/rest/v1/`, `/auth/v1/`, etc.)
- **Gateway Port** — `:8000` (configurable via `SHARED_GATEWAY_PORT`)
- **Client SDKs** — Supabase JS/Python clients work without changes
- **Environment** — `SHARED_KONG_HTTP_PORT` renamed to `SHARED_GATEWAY_PORT`
