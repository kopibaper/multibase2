## Feature 3 — Log Drains

### Platzierung

**Neuer Tab `log-drains`** in `WorkspaceProjectPage.tsx`, nach `replicas`.

### Ziel

Log Drains leiten Docker-Container-Logs einer Instanz an externe HTTP-Webhook-Endpoints weiter
(z.B. Datadog, Logtail, eigener Logserver). Das Backend pollt alle 30 Sekunden die neuesten
Log-Zeilen und postet sie als JSON.

### Backend Tasks

**Datei:** `dashboard/backend/prisma/schema.prisma`

```
Task BE-3.1: LogDrain-Modell hinzufügen (siehe Schema oben)
  - Instance @relation: logDrains LogDrain[]
  - Migration: prisma migrate dev --name add_log_drains
```

**Neue Datei:** `dashboard/backend/src/services/LogDrainService.ts`

```
Task BE-3.2: LogDrainService
  constructor(prisma, dockerManager)

  start(): void
    → setInterval(this.deliverAll, 30_000)

  private async deliverAll(): Promise<void>
    → alle enabled LogDrains laden
    → pro Drain: getLogs(drain) → postLogs(drain, lines)

  private async getLogs(drain): Promise<string[]>
    → dockerManager.listProjectContainers(instanceName)
    → filtert nach drain.services (split(','))
    → dockerManager.getContainerLogs(containerId, { tail: 200, since: Math.floor(Date.now()/1000 - 35) })
    → gibt Array von { timestamp, service, message } zurück

  private async postLogs(drain, lines): Promise<void>
    → POST drain.url mit axios (Timeout 10s)
    → Body: { source: 'multibase', instance: instanceName, drain: drain.name, lines }
    → Header: Content-Type: application/json
    → Aktualisiert drain.lastStatus + drain.lastDelivery in DB
    → Bei Fehler: status auf Fehler-Code setzen, kein Crash
```

**Neue Datei:** `dashboard/backend/src/routes/log-drains.ts`


## Feature 5 — MCP Server

### Platzierung

Globale Settings-Seite (keine Workspace-Tab-Instanz). Neue Route `/settings/mcp` im Frontend.  
Backend: `/api/mcp` (MCP HTTP-Transport, streamable HTTP).

### Ziel

Ein MCP-kompatibler Server, der Multibase-Instanz-Management als KI-Tools bereitstellt.
AI-Assistenten (Claude Desktop, Cursor, VS Code Copilot) können damit Multibase direkt steuern.

### Protokoll

MCP Streamable HTTP Transport (`2025-03-26`-Spezifikation):
- `POST /api/mcp` — JSON-RPC Request (tools/call, tools/list, resources/list, prompts/list)
- `GET /api/mcp/sse` — Server-Sent Events (optional, für streaming responses)
- Auth: `Authorization: Bearer <API_KEY>` (bestehender API-Key-Mechanismus)

### MCP Tools (Übersicht)

| Tool Name | Beschreibung |
|-----------|--------------|
| `list_instances` | Alle Multibase-Instanzen auflisten mit Status |
| `get_instance` | Details einer einzelnen Instanz (Status, Ports, Labels) |
| `start_instance` | Instanz starten |
| `stop_instance` | Instanz stoppen |
| `restart_instance` | Instanz oder einzelnen Service neu starten |
| `execute_sql` | SQL auf einer Instanz ausführen (SELECT-only by default) |
| `get_logs` | Container-Logs einer Instanz/Service abrufen |
| `get_health` | Health-Status einer Instanz abrufen |
| `get_metrics` | CPU/RAM-Metriken einer Instanz abrufen |
| `create_instance` | Neue Instanz erstellen (name, environment) |
| `backup_instance` | Backup einer Instanz auslösen |
| `list_functions` | Edge Functions einer Instanz auflisten |

### Backend Tasks

**Neue Datei:** `dashboard/backend/src/routes/mcp.ts`

```typescript
// MCP Streamable HTTP Transport — JSON-RPC 2.0

Task BE-5.1: createMcpRoutes(instanceManager, dockerManager, metricsCollector, prisma)

POST /
  - Header-Check: Authorization: Bearer <key>   (nutzt existierenden apiKeyAuth)
  - Body: JSON-RPC 2.0 Request { jsonrpc, id, method, params }
  - method === 'initialize':
      → { protocolVersion: '2025-03-26', capabilities: { tools: {} }, serverInfo: { name: 'multibase-mcp', version: '1.7.0' } }
  - method === 'tools/list':
      → Gibt Array alle Tool-Definitionen (name, description, inputSchema) zurück
  - method === 'tools/call':
      params: { name: string, arguments: Record<string, unknown> }
      → Dispatch zu internem Handler basierend auf name
      → Gibt { content: [{ type: 'text', text: JSON.stringify(result) }] } zurück
  - Fehler → { error: { code, message } } mit passenden JSON-RPC Error Codes
  - Response: Content-Type: application/json (kein SSE erforderlich für basic tools)

GET /info
  - Gibt MCP-Server-Infos zurück: endpoint URL, verfügbare Tools, Version
  - Kein Auth erforderlich — für Dashboard-Anzeige
```

**Neue Datei:** `dashboard/backend/src/services/McpService.ts`

## Technische Hinweise

### CodeMirror (Feature 1)
- `@uiw/react-codemirror` und `@codemirror/theme-one-dark` und `@codemirror/lang-html` sind bereits installiert
- Nur `@codemirror/lang-javascript` (für TypeScript-Support) muss neu installiert werden
- `javascript({ typescript: true })` aktiviert sowohl JS als auch TS Highlighting

### Read Replicas (Feature 2)
- Die Replica-URL wird niemals im Frontend im Klartext zurückgegeben (maskieren: zeige nur `postgresql://***@host:port/db`)
- Security: Der Backend-Service verbindet sich **direkt** mit der Replica-URL — diese darf nur von vertrauenswürdigen Hosts erreichbar sein
- Verbindungstest mit `pg.Client` und `connect_timeout = 5` Sekunden

### Log Drains (Feature 3)
- URL-Validierung: nur `https://` in Production (warnen bei `http://`)
- Rate Limiting: max 3 Drains pro Instanz (verhindert Spam an externe Dienste)
- Der `since`-Parameter nutzt Unix-Timestamp der letzten Lieferung um Duplikate zu vermeiden
- Empfehlung: `LogDrainService.lastDeliveryTime` als Map im Memory halten (pro drainId)

### Realtime (Feature 4)
- Container-Name: `realtime-dev.<instanceName>-realtime` (aus InstanceManager.ts bekannt)
- Neustart des Containers via `dockerManager.restartContainer(containerId)` — benötigt den Container-ID
- `pg_stat_replication`-Tabelle existiert nur auf dem Primary — nie auf einer Replica abfragen

### MCP Server (Feature 5)
- `execute_sql` ist auf SELECT/WITH/EXPLAIN beschränkt (read-only guard im McpService)
- **Keine** sensiblen Werte (JWT Secrets, Passwörter) in Tool-Responses zurückgeben
- Schema-Validierung der Tool-Arguments mit `zod` (bereits als Dependency vorhanden)
- Der MCP-Endpoint ist öffentlich via nginx weitergeleitet — API-Key-Pflicht ist Sicherheitskritisch
- Rate Limiting: MCP-Endpoint sollte mit `express-rate-limit` (bereits vorhanden) abgesichert werden:
  `max: 100 requests per minute per API key`
